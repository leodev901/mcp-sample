# AGENTS.md

## Language
- 모든 답변과 설명은 한국어로 작성한다.
- 코드 식별자(변수/함수/클래스)는 영어로 작성한다.
- 영어 용어를 쓸 때는 한국어 한 줄 설명을 함께 제공한다.
- TASK와 PLANNING, REVIEW를 비롯하여 생각하는 과정을 보여 줄때도 한국어로 작성한다.

### Thinking
- 질문을 섣불리 판단하여 단편적인 답변/수정을 하지 않는다. 
- 사전에 전제조건과 연관된 전체 코드 맥락을 한 번에 확실하게 검토한 뒤,정확도 높은 결과물만 제공해 드리겠습니다.
- 절대 해선 안되는 것이 '오답'으로 인한 나의 '지적'과 '수정' 요처이 반복되며 쓸데 없이 토큰이 소모됨으로써 비용이 발생하는 것이다. 지금 비싼 요금으로 토큰을 사용하는 만큼, 시간이 오래걸리더라도 '한 번에 정확한 대답'을 해주것을 보장해라 


## Learning Mode (학습모드)
- 기본 모드는 학습모드다.
- 결과를 제시할 때 순서를 지킨다: 문제 정의 -> 접근 방법 -> 코드 -> 검증 -> 한 줄 요약.
- 실습 코드에 문법에 대한 설명을 가미한다.
- "왜 이렇게 했는지"를 1~3문장으로 설명한다.
- 가능하면 대안 1개와 트레이드오프 1개를 함께 제시한다.

## Coding Style
- 기존 프로젝트 스타일과 린트 규칙을 우선한다.
- 암묵적 동작보다 명시적 코드를 선호한다.
- 복잡한 로직에는 짧은 의도 주석을 추가한다.
- 불필요한 추상화와 과도한 최적화는 피한다.

## Comments (주석 규칙)
- 모든 작성 코드에는 주석이 포함되어야 한다.
- 주석은 초보자도 이해 할수 있어야 하며 '문법'에대한 설명그리고 "왜"를 설명한다.
- 함수가 길거나 조건 분기가 복잡하면 블록 상단에 1~2줄 요약 주석을 작성한다.

## Guide Docs Output
- 기능/리팩토링/버그 수정 시 관련 가이드 문서를 함께 갱신한다.
- 가이드 문서 파일명은 `OOO_GUIDE.md` 규칙을 사용한다.


## Diagram Rules
- 아키텍처/흐름 설명이 필요하면 Mermaid 또는 asrcii-arg 문법을 사용한다.
- 다이어그램 아래에 텍스트로 3~5줄 설명을 추가한다.
- 다이어그램과 실제 코드 경로를 함께 명시한다.

## Example Rules
- 예시는 복붙 실행 가능해야 한다.
- 예시마다 전제조건(버전/환경변수)과 기대 결과를 적는다.
- 가능하면 실패 예시 1개와 해결 방법을 함께 제공한다.


## Output Format
- 파일 변경 시 경로를 명시한다.
- 긴 설명보다 실행 가능한 단계와 명령어를 우선한다.
- 요청이 단순하면 짧게, 학습 요청이면 구조적으로 자세히 설명한다.
